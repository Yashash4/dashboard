## CRITICAL

|----|-----------|-------------|----------|
| C-01 | `src/lib/provision-v3.ts:467-468` | Synchronous `require()` inside async provisioning function | `const dataApiToken = require("crypto").randomBytes(32).toString("hex");` and `const dataApiFiles = require("./vps-data-api-bundle").getBundle(dataApiToken);` use dynamic `require()` at runtime. The `vps-data-api-bundle` module is loaded via `require()` which bypasses tree-shaking and may fail in edge runtime. More critically, the generated `dataApiToken` is never saved to the database — it is computed on line 467 but the `provisionVPS` function returns `{ success: true }` on line 496 without including `dataApiToken` in the result, and `route.ts` line 266-303 never writes a data API token to `vps_instances`. If the DB write fails, the job is marked as successful (`completeJob` on line 305) even though the VPS record was not saved. The admin sees "success" but the customer has no VPS in the database, requiring manual DB intervention. |
| C-03 | `src/app/api/admin/provision/route.ts:78-88` | Background async function errors are silently swallowed in edge cases | `runProvisioning()` is called without `await` (fire-and-forget). If `runProvisioning` throws synchronously before entering the try block (e.g., if `createSubdomain` import is broken), the unhandled promise rejection will crash silently with no job status update. While the function has a try/catch, any error in the catch block itself (e.g., `completeJob` or `logAudit` throwing) creates an unhandled rejection with the job stuck in "running" state forever (until the 10-minute cleanup that only fires after `completeJob` is called — which never happens). |

| H-01 | `src/lib/provision-v3.ts:180` | Base64 string injected with single quotes can break shell commands | On line 180, `echo '${configB64}' | base64 -d > ...` uses single quotes around the base64 string. While standard base64 doesn't contain single quotes, the same pattern is used throughout (lines 284, 383, 445, 462, 474). If any base64 variant produces a `'` character, the shell command breaks. The `base64url` encoding used for `dashboardPassword` in `route.ts:246` does NOT go through this path, but the `Buffer.from().toString("base64")` standard encoding alphabet (`A-Za-z0-9+/=`) is safe. The DB says "stopped" but the VPS continues running. This is a billing integrity issue — suspended customers continue to consume resources. Compare with `customers/[id]/actions/route.ts` which does SSH to run `systemctl stop`. |
