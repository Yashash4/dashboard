## CRITICAL

| ID | File:Line | Description | Evidence |
|----|-----------|-------------|----------|
| C1 | `src/app/page.tsx:1` | **Landing page is "use client" — destroys SSR and SEO.** The root page component is marked `"use client"`, which means the entire landing page is client-rendered. Search engine crawlers receive an empty `<main>` shell with no content. All text (hero, features, pricing, FAQ, comparison) is invisible to crawlers until JS executes. This is the single most damaging SEO issue on the site. | Line 1: `"use client";` — combined with `dynamic(() => ..., { ssr: false })` on lines 10-15, the below-fold sections are not even hydrated server-side. The page exports no metadata (no `export const metadata`), so the landing page relies entirely on `layout.tsx` metadata. |
| C2 | `src/app/page.tsx:10-15` | **Six sections loaded with `ssr: false` — zero server-rendered HTML for most of the page.** Features, Pricing, HowItWorks, Comparison, FAQ, and CTA all use `dynamic(() => import(...), { ssr: false })`. This means their HTML is completely absent from the initial server response. Googlebot and social media scrapers see only Hero + Navbar + ChannelBar + Footer. All pricing information, FAQ structured data candidates, feature descriptions, and comparison tables are invisible. | `const Features = dynamic(() => import("@/components/landing/Features"), { ssr: false });` repeated for 6 components. |

## HIGH

This is the slowest way to load fonts — it blocks rendering until the CSS is fetched and parsed, then makes a second network request to Google Fonts. Should use `next/font` for self-hosted, zero-layout-shift font loading. | `@import url('https://fonts.googleapis.com/css2?family=DM+Sans:...&family=Instrument+Serif:...&family=JetBrains+Mono:...')` — three families with multiple weights. |

For anonymous visitors (the majority of landing page traffic), this is a wasted round-trip to Supabase's auth endpoint. It delays interactive CTA rendering (button text depends on `user` state) and adds ~100-300ms latency. | Hero line 20-24: `createBrowserClient(...)` then `supabase.auth.getUser()`. Navbar lines 92-97: identical pattern. Two separate Supabase client instances making the same auth call on every page load. |

With `"use client"`, this is impossible without a `generateMetadata` function in a server wrapper. | `layout.tsx` line 5: `export const metadata: Metadata = { ... }` is the only metadata source.
