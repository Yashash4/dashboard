## Summary of Systemic Patterns

### 1. Missing Error Checks on DB Writes
Multiple routes perform DB writes (insert/update/upsert) without checking the `error` return value. The most critical instances are in provisioning (C-01), channel routing (M-03), thread messages (H-07), and idempotency cache (M-09). This pattern means failures are silently ignored and the user receives a success response when data was not actually persisted.

### 2. Unbounded Queries
Several endpoints fetch all matching rows without `.limit()`, including high-traffic tables like `agent_analytics` (C-04, H-01), `chat_messages` (H-02), and all "list" endpoints for user-created resources (L-01 through L-06). The most dangerous is C-04 which is accessible via the public V1 API.

### 3. Non-Atomic Multi-Step Operations
Payment processing (C-02), account deletion (H-03), task reordering (M-05), and VPS provisioning DB writes (C-01) all perform multiple dependent DB operations without transactions. Supabase's JS client does not support client-side transactions, so these should use Postgres functions (RPCs) for atomicity.

### 4. In-Memory Rate Limiter is a No-Op
The synchronous `rateLimit()` function (H-08) that most routes rely on never records hits, meaning it will never actually rate-limit anyone. Only `rateLimitAsync()` (which calls the DB-backed RPC) would work, but it is not used anywhere in the codebase. All rate limiting in the application is effectively non-functional.

### 5. N+1 Query Patterns
The circular dependency check (H-05) and review approval cascade (H-06) both issue sequential per-row DB queries inside loops. These should be refactored to use batch queries or recursive CTEs.

### Recommended Indexes (inferred from query patterns)
- `agent_analytics(api_key_id, created_at)` -- used by V1 usage and key stats queries
- `agent_analytics(user_id, agent_id, created_at)` -- used by chat analytics
- `chat_messages(conversation_id, role)` -- used by funnel analysis and intent classification
- `mc_task_dependencies(task_id)` and `mc_task_dependencies(depends_on_task_id)` -- used by dependency graph traversal
- `api_threads(user_id, updated_at)` -- used by thread listing
- `idempotency_cache(key, user_id, expires_at)` -- used by idempotency checks
- `rate_limits(identifier, window_start)` -- documented in schema comment but should be verified as existing
